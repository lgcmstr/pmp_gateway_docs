Introduction

Welcome to the PMP Gateway API documentation. This API reference provides information on available endpoints and how to interact with them.

Using the PMP Gateway API, you can integrate patient prescription history data into your applications.

This API is RESTful and uses XML formatted requests and responses.

Questions & Feedback

This documentation and all example code is available on GitHub. If you have questions or feedback the best way to contact us is to open an "issue" on our GitHub project space.

Example code is provided for each API request. To get started, you can make requests against the testing environment using the code examples and the XML request data they reference.

Authentication

You must authenticate with a username and password for all API requests.

All requests must be made over HTTPS. Calls made over plain HTTP will fail.

Your username and password is provided to you by the Appriss Project Manager who is handling your account.

API Endpoints

A testing environment is available for testing and development purposes. The testing environment is populated with test data and contains no real-world patient data.

The production environment contains real HIPAA protected patient data and must not be used for testing.

Testing API Endpoint

https://gateway-prep.pmp.appriss.com

Production API Endpoint

https://pmpgateway.net

Score Request

Schema files:
score_request_schema.xsd
request_constraints.xsd

If prescription data is found for the requested patient, the response will include:

  • Risk scores for three prescription categories:
    • narcotics
    • sedatives
    • stimulants
  • A URL to request a NARxCHECK report for the patient.

The score request has many optional data elements. Additionally, some elements are conditional, meaning that your request must include at least one element out of a set of possible elements.

This example includes all required elements and most of the optional elements. For conditional elements, this example includes all possible options. There are notes where conditional elements are present.

Note: Some optional elements have been ommitted because they are not useful for the vast majority of users. For more information on these rarely used elements, consult the score request XML schema.

Definition

POST https://gateway-prep.pmp.appriss.com/narx/request/simple

Example Request

curl https://gateway-prep.pmp.appriss.com/narx/request/simple \
  -u username:password \
  -H "Content-Type: application/xml" \
  -H "Accept: application/xml" \
  -d @score_request.xml

Example Request Data

score_request.xml

download 
<?xml version="1.0" encoding="UTF-8"?>
<NarxScoreRequest xmlns="http://xml.appriss.com/gateway">
  <Requester>
    <SenderSoftware><!-- Optional -->
      <Developer>Software Inc.</Developer>
      <Product>Product Name</Product>
      <Version>Version info</Version>
    </SenderSoftware>
    <RequestDestinations><!-- Optional -->
      <Pmp>KS</Pmp>
    </RequestDestinations>
    <Provider>
      <Role>Physician</Role>
      <FirstName>John</FirstName>
      <LastName>Doe</LastName>
      <!-- At least one identifier is required: NPINumber, DEANumber, or ProfessionalLicenseNumber. -->
      <DEANumber>AB1234579</DEANumber>
      <NPINumber>1212345671</NPINumber>
      <ProfessionalLicenseNumber>
        <Type>Something</Type>
        <Value>A1231xbzy-bx</Value>
        <StateCode>KY</StateCode>
      </ProfessionalLicenseNumber>
    </Provider>
    <Location>
      <Name>Store #123</Name>
      <!-- At least one identifier is required: NPINumber, DEANumber, or NCPDPNumber. -->
      <DEANumber>AB1234579</DEANumber>
      <NPINumber>1234567890</NPINumber>
      <NCPDPNumber>1234567</NCPDPNumber>
      <Address><!-- Only StateCode is required -->
        <Street>Street1</Street>
        <Street>Street2</Street>
        <City>City</City>
        <StateCode>OH</StateCode>
        <ZipCode>40242</ZipCode>
        <ZipPlusFour>4242</ZipPlusFour>
      </Address>
    </Location>
  </Requester>
  <PrescriptionRequest>
    <Patient>
      <Name>
        <First>Bob</First>
        <Last>Testpatient</Last>
      </Name>
      <Birthdate>1900-01-01</Birthdate><!-- YYYY-MM-DD -->
      <SexCode>M</SexCode><!-- Optional -->
      <!-- ZipCode or Phone is required -->
      <Address>
        <City>Wichita</City>
        <StateCode>KS</StateCode>
        <ZipCode>67203</ZipCode>
      </Address>
      <Phone>1234567890</Phone>
      <!-- Optional - Provide the Medical Record ID to display historical scores in reports -->
      <MedicalRecordID>any-string-id</MedicalRecordID>
    </Patient>
  </PrescriptionRequest>
</NarxScoreRequest>

Errors

NARxCHECK uses conventional HTTP response codes to indicate the success or failure of an API request. In general, codes in the 2xx range indicate success, codes in the 4xx range indicate an error that resulted from the provided information (e.g. a required parameter was missing, etc.), and codes in the 5xx range indicate an error with the NARxCHECK servers.

HTTP Status Codes

200 OK Everything worked as expected or the request was missing required XML nodes and the error will be described in the XML response.
401 Unauthorized Invalid login credentials provided.
500 Internal Server Error Something went wrong. The response body will be a NARxCHECK Response with an Error section.

Report Request

Schema files:
report_request_schema.xsd
request_constraints.xsd

To make a report request, you must first make a successful score request. The response to the score request will include the URL where you send report requests.

Report Details

If the report request is successul, the response will include a NARxCHECK report.

NARxCHECK reports can be returned in two ways. You specify the method by including or excluding the <ReportLink /> element from your request. If you exclude it, the report will be returned as raw HTML. If you include it, the response will include a URL where the report can be accessed once.

Request Data

The report request has many optional data elements. Additionally, some elements are conditional, meaning that your request must include at least one element out of a set of possible elements.

This example includes all required elements and most of the optional elements. For conditional elements, this example includes all possible options. There are notes where conditional elements are present.

Note: Some optional elements have been ommitted because they are not useful for the vast majority of users. For more information on these rarely used elements, consult the report request XML schema.

  -u username:password \
  -H "Content-Type: application/xml" \

Errors

NARxCHECK uses conventional HTTP response codes to indicate the success or failure of an API request. In general, codes in the 2xx range indicate success, codes in the 4xx range indicate an error that resulted from the provided information (e.g. a required parameter was missing, etc.), and codes in the 5xx range indicate an error with the NARxCHECK servers.

HTTP Status Codes

200 OK Everything worked as expected.
400 Bad Request Often missing a required parameter.
401 Unauthorized Invalid login credentials provided.
404 Not Found The requested resource doesn't exist.
410 Gone The report has expired.
500 Internal Server Error Something went wrong with NARxCHECK. The response body will include information on why it failed.
504 Gateway Timeout The report is not yet generated.
